# Dispatch v1.1 — Repricing/Relisting Engine + Task Dashboard

**Date:** 2026-07-10
**Status:** Approved (brainstorming session with Wil, all sections reviewed)
**Builds on:** v1 spec (`2026-07-10-dispatch-v1-design.md`), shipped on main.

## What v1.1 is

Dispatch starts telling Wil what needs attention. A pure derivation engine computes due tasks from existing data every time the app loads; a Today's Tasks section on the inventory page presents them with one-click actions. Nothing changes without a click (human-assisted philosophy), and nothing runs in the background (local-first, no scheduler).

## Design decisions (from brainstorm)

1. **Relisting rules belong to channels, not items.** Each publisher module declares a `relistPolicy`. Researched policies (July 2026):

| Channel | Method | Suggested cadence | Hard platform rule |
|---|---|---|---|
| Reddit r/Watchexchange | repost (new [WTS] thread) | every 7 days | subreddit forbids repost inside 7 days |
| OfferUp | delete + fresh post | every 7 days | none (freshness algorithm) |
| Facebook Marketplace | renew, then fresh relist | renew every 7 days; fresh relist after 42 days | renew 1×/week, max 5 per listing |
| Craigslist | renew, then fresh repost | renew every 3 days; fresh repost after 30 days | 48h minimum between renews/identical posts (flagging risk) |
| Watchuseek | bump/edit thread | every 7 days | forum etiquette (conservative default) |

Policy shape: `{ method: "repost" | "delete-repost" | "renew-then-repost" | "bump", intervalDays, minIntervalDays, freshRelistAfterDays? }`. The engine suggests at `intervalDays` and never inside `minIntervalDays`.

Sources: [Watch Trading Academy on r/Watchexchange](https://www.watchtradingacademy.com/general/watch-traders-guide-to-buying-and-selling-on-reddit/), [OfferUp support](https://help.offerup.com/hc/en-us/articles/360032335951-Update-an-item-listing), [Voolist OfferUp guide](https://www.voolist.com/blog/offerup-selling-guide-2026), [Facebook renew limits](https://mbial.com/how-many-times-can-you-renew-a-facebook-marketplace-listing/), [Crosslist on FB relisting](https://crosslist.com/blog/how-to-relist-on-facebook-marketplace), [Craigslist renew help](https://www.craigslist.org/about/help/posting/modify/renew), [Craigslist repost help](https://www.craigslist.org/about/help/repost).

2. **Pricing cadence is per-item custom.** Dollar OR percent steps (setting both is rejected), on an interval measured from the last price change. Floor is the item's minimum price, always.
3. **Suggest + one-click apply.** A due drop appears as a task with the computed target price; clicking Apply updates the asking price and logs history. The engine never changes data on its own.
4. **Derived on page load.** `computeTasks(items, listings, priceHistory, publishers, now)` is a pure function; no task table, no scheduler, no background process. The dashboard is correct the moment it renders.
5. **Snooze per item.** One `snoozed_until` timestamp suppresses all of an item's tasks; default snooze +7 days; snoozed items show 💤 on their inventory card.

## Task types (all derived)

1. **Price drop due** — published item with cadence; `now − lastPriceChange ≥ drop_interval_days`; asking > minimum; not snoozed. Target = asking − (drop_amount | drop_percent), rounded to the nearest whole dollar, clamped to minimum. `lastPriceChange` = latest `price_history.changedAt`, falling back to the item's `createdAt` when no history row exists.
2. **Relist due (per channel)** — active listing whose age (or time since `renewed_at`) ≥ channel policy interval; only where policy permits; respects `minIntervalDays` and `freshRelistAfterDays`; not snoozed.
3. **Stale listing price** — active listing whose `listedPrice ≠ item.askingPrice`. Naturally appears right after applying a drop, closing the loop.
4. **Ready to publish** — item with status `ready` and zero active listings; not snoozed.

## Data model changes

Four nullable columns on `items`: `drop_amount` (real), `drop_percent` (real), `drop_interval_days` (integer), `snoozed_until` (text date). One nullable column on `listings`: `renewed_at` (text date). No new tables.

Consequence: `needs_repricing` / `needs_relisting` statuses are obsolete — "needs attention" is computed, never stored. They are hidden from status pickers; no schema migration (existing rows never used them).

## Architecture

```
Dashboard (Today's Tasks section, inventory page top)
   ↓ calls on load
Engine: computeTasks(...) — pure, src/lib/tasks.ts, injectable now
   ↓ reads
items + listings + price_history          publishers[].relistPolicy
```

Actions (server): `applyPriceDrop(itemId)` (update asking, log history), `confirmListingUpdated(listingId)` (sync listedPrice to asking, stamp renewed_at — serves both stale-price confirmations and "I renewed it" on renew-method channels), `snoozeItem(itemId, days=7)`. The existing end-listing action is reused by relist tasks ("End old + open Publish" lands on the Publish tab).

## UI

- **Today's Tasks** section above the item grid: one card per task with the specific one-click action + Snooze. Section absent when no tasks.
- **Item form** gains one optional row: "Price drops: [$ or %] every [N] days" (enabling nudges toward setting a minimum price).
- **Publish tab** shows each channel's relist policy inline.
- **Inventory cards** show 💤 while snoozed.

## Edge cases

- Drop below minimum → clamp to exactly minimum (final drop); at minimum → no further drop tasks.
- No cadence → no drop tasks. Cadence without minimum → uncapped drops (user's explicit setup choice; form nudges).
- Draft/sold/archived items → no tasks; `ready` participates only in ready-to-publish.
- Facebook renew ceiling handled by listing age (42 days → suggest fresh relist), no renew counting.
- Craigslist 48h floor enforced via `minIntervalDays` regardless of snooze timing.

## Explicitly not in v1.1

Automation (v1.2), notifications/background scheduler, policy-editing UI (defaults live in publisher modules), per-channel pricing.

## Verification

1. **Engine unit tests** — pure function with explicit `now`: fixtures for each task type and every edge case above (clamping, rounding, $-vs-% precedence, snooze suppression, min-interval enforcement, 42-day Facebook switch).
2. **Action tests** — apply-drop / confirm-updated / snooze against in-memory SQLite with real migrations (pattern established by photo-order tests).
3. **Browser acceptance** — cadence on a real item, backdated history to simulate age, full loop: drop task → Apply → stale-price task → confirm → clean dashboard; snooze hides tasks and shows 💤.
