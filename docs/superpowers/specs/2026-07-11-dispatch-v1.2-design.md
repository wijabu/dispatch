# Dispatch v1.2 — Human-Assisted Publishing

**Date:** 2026-07-11
**Status:** Approved (brainstorming session with Wil, all sections reviewed)
**Builds on:** v1 (publishers + Publish tab + ledger) and v1.1 (task engine), both shipped on main.

## What v1.2 is

Dispatch starts doing the typing. Two tiers of assistance, chosen per channel by risk profile — and on every tier, **Dispatch never clicks Submit**: you review, you post, you confirm with the existing one-click "Mark listed."

| Tier | Channels | What one click does |
|---|---|---|
| **Auto-fill** | Craigslist, Watchuseek, OfferUp | Playwright opens the posting form in Dispatch's own browser profile, types the generated listing, uploads the photos from disk, then detaches — the window is yours to review and submit. |
| **Staged handoff** | Facebook (full), Reddit (lite) | Listing text → clipboard; posting page opens in a dedicated non-automated browser; photos revealed in Finder ready to drag (FB). You paste, drag, post. Zero automation fingerprint. |

## Design decisions (from brainstorm)

1. **Channel selection is a risk decision.** Auto-fill only where automation risk is acceptable: Craigslist (low), Watchuseek (low-moderate), OfferUp (moderate). Facebook's fingerprint-based automation detection makes any DOM automation there an account risk — even "just filling fields" — so FB is staged-handoff only.
2. **You submit, you confirm.** After filling, Dispatch detaches completely; no auto-submit, no submission detection. The marketplace's Submit is yours; "Mark listed" in Dispatch (price pre-filled) remains the ledger confirmation.
3. **Login in the flow, sessions in a profile, passwords nowhere.** The Playwright browser uses a persistent profile at `data/browser-profile/` (gitignored). First "Open & pre-fill" per channel hits the login wall → Dispatch detects it, leaves the window open, says "log in there, then retry." Session persists thereafter. Deleting the profile directory wipes all Tier-1 sessions.
4. **Architecture A — sibling layer, in-process browser.** New `src/automation/` directory; publishers stay 100% pure (`generate()` + `relistPolicy`, unchanged). Fill scripts consume `generate()`'s output. The browser is a module-level singleton (`chromium.launchPersistentContext`, headed) in the Next server process — same pattern as the db singleton. A future daemon (approach B) would reuse the fill scripts unchanged; build it only if window-lifetime pain appears.
5. **Facebook staging opens in Firefox** (Wil's pick; configurable in `src/config/staging.ts`). Rationale: sessions persist (log in once — unlike incognito, which forces re-login + 2FA checkpoints every time), FB's cookies are quarantined from the daily browser so the FB pixel on third-party sites can't link his browsing, and Firefox's tracking protection layers on top. Honest limit, documented in-app copy not required: while logged in on FB, server-side tracking of on-FB activity is unaffected by any browser choice.
6. **Reddit stays un-automated in v1.2, deliberately.** (a) [WTS] posts require externally hosted photos (Imgur album + timestamp shot) — form-filling automates the easy 20%; (b) Reddit aggressively fingerprints browser automation and shadowbans, uniquely costly on rep-based WTS subs; (c) the sanctioned path is Reddit's official OAuth API. **Named v1.3 candidate: Reddit posting via official API + Imgur API photo hosting.** Meanwhile Reddit gets lite staged handoff: copy bundle + open the submit page.

## The fill contract

```ts
interface FillScript {
  publisherId: string;              // "craigslist" | "watchuseek" | "offerup"
  startUrl: string;                 // posting form entry point
  isLoginWall(page: Page): boolean; // detects redirect-to-login
  fill(page: Page, ctx: FillContext): Promise<FillResult>;
}

interface FillContext {
  listing: GeneratedListing;        // from the publisher's pure generate()
  item: Item;
  photoPaths: string[];             // absolute paths, primary photo first
}

type FillResult =
  | { status: "ready" }                                        // review & submit
  | { status: "login_required" }                               // log in, retry
  | { status: "partial"; failedAt: string; filled: string[] }  // e.g. photos failed
  | { status: "failed"; reason: string };                      // window open at form; copy/paste
```

- Registry in `src/automation/index.ts` keyed by publisher id, mirroring the publisher registry.
- Photos upload via Playwright `setInputFiles` with the item's local photo paths in sort order (primary first).
- Each field fills inside its own try/catch — one broken selector never aborts the rest of the form.

## Degradation ladder (channel-card feedback)

| Outcome | Card shows |
|---|---|
| ✅ ready | "Ready — review the window and submit." |
| 🔐 login_required | "Log in in the open window, then hit Pre-fill again." |
| ⚠️ partial | "Filled title/price/description; photo upload failed — add photos by hand." Window open where it stopped. |
| ❌ failed | "Auto-fill failed (page changed?) — window is open at the form; use the copy buttons." |

v1's copy/paste tier is the permanent floor; it never goes away.

## Lifecycle rules

- **One fill at a time** — a simple in-process lock; the button disables while a fill runs (10–30s with photos).
- **After the fill, the window is yours** — Dispatch detaches; it never clicks Submit and never watches the page.
- **Windows live as long as the app** — restarting Dispatch closes open filled windows; re-run Pre-fill.
- **Per-channel kill switch** — a config flag disables auto-fill for a channel (falls back to staged handoff/copy) if a marketplace ever pushes back.

## Staged handoff details

- **Facebook (full):** one click → clipboard gets the generated title/description/price as a paste-ready bundle (client-side) → `open -a Firefox "https://www.facebook.com/marketplace/create/item"` (server action) → photos revealed in Finder via `open -R` (server action). Deep links ride along: FB stale-price and renew tasks get an **"Open listing"** button using the ledger's stored URL, opened in the same dedicated browser, with the new price on the clipboard.
- **Reddit (lite):** one click → clipboard gets `[WTS]` title + body → the r/Watchexchange submit page opens in your default browser (client-side `window.open`; no quarantine requested for Reddit). Photo hosting stays manual until the v1.3 API work.

## UI changes

- **ChannelCard** gains one button per tier: "Open & pre-fill" (Tier 1) or "Stage for Facebook/Reddit" (Tier 2), alongside the existing copy buttons; fill status renders on the card per the degradation ladder.
- **TaskSection**: Facebook stale-price and renew task rows gain an "Open listing" button (ledger URL → dedicated browser, new price on clipboard). No other task-row changes.
- Task engine logic, dashboard grid, and ledger actions are untouched.

## New dependency

`playwright` (runtime dep) + its Chromium download (`npx playwright install chromium`). The launcher/build docs gain that one setup step. Next config may need `serverExternalPackages: ["playwright"]`.

## Explicitly not in v1.2

No DOM automation on Facebook or Reddit; no auto-submit or submission detection; no background/scheduled fills; no credential storage ever; no separate browser daemon; no Imgur/photo-hosting integration (v1.3 candidate with Reddit API).

## Testing

1. **Fixture pages** — simplified HTML replicas of each marketplace's posting form, committed to the repo. Playwright tests run each fill script against its fixture: text fields receive the generated values, the file input receives photo paths primary-first, `isLoginWall` fires on a fixture login page.
2. **What fixtures can't prove** — the real sites' DOM. By design: real-site breakage is a runtime event the degradation ladder handles; CI does not pretend to catch marketplace redesigns.
3. **Existing suite stays green** — publishers and the engine are untouched (83 tests).
4. **Live acceptance** — one guided real posting per channel by Wil: Craigslist first (posts delete instantly), then Watchuseek, then OfferUp. FB staging verified end-to-end (clipboard, Firefox open, Finder reveal).

## Risk register

| Risk | Posture |
|---|---|
| Selector rot | Will happen; ladder degrades to copy/paste; a fill-script fix is small and isolated. |
| OfferUp bot detection | Moderate; headed Chromium + persistent profile + human submit is the gentlest posture; captchas solved by hand; kill switch if warnings appear. |
| Craigslist / Watchuseek | Minimal / etiquette-only. |
| Facebook & Reddit accounts | Zero new risk — no automation touches either. |
| ToS gray zone | Acknowledged; human-review-and-submit on every post, personal scale. Wil's informed call. |
