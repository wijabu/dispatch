# Dispatch v1.4 — Facebook Marketplace automation (design)

Date: 2026-07-18
Status: approved design, pre-implementation

## Goal

Bring Facebook Marketplace up to the same hands-off automation Dispatch already
gives OfferUp: **post**, **reprice**, and **relist** driven from the Facebook
Android app in the emulator. Facebook is the second of the two channels on Wil's
success bar ("remove the friction of drafting listings and of updating each app
to drop price or relist"); OfferUp shipped in v1.3, this closes the pair.

## Why the emulator (not the browser)

v1.2 deliberately made Facebook a **staged handoff** because Facebook's
fingerprint-based detection makes DOM automation of its *website* an account
risk even just filling fields. That decision stands — the web is out. The new
option is the Facebook Android app in the emulator, the same surface that made
OfferUp work. Wil accepted the associated risk (a brand-new account can be
checkpointed/banned) in exchange for full hands-off automation.

## Feasibility probe (done 2026-07-18 — GREEN)

Before committing, we ran the probe live on the existing AVD (`emulator-5554`):

- Installed the main Facebook app (`com.facebook.katana`) and **logged in** with
  the new account — this is where it most likely dies (Facebook often refuses
  login on emulators); it stuck.
- Reached Marketplace → Seller Hub → **Create listing → One item** and **posted a
  real listing by hand**. It published; the Seller Hub shows **1 active listing**.
  No checkpoint / "confirm it's you" / posting-limit wall.
- The composer is **cleanly addressable by uiautomator** — semantic resource-ids
  `composer_v3_title`, `composer_v3_price`, `composer_v3_description`, plus
  `Add photos`, a `Condition` chip row, and a category control. This was the
  make-or-break unknown (native Android could have been an opaque canvas); it's
  as clean as OfferUp's testIDs.
- The Seller Hub **natively surfaces the maintenance concepts** we need:
  "Listings to renew" and "Listings to delete & relist" tiles, under
  `mp_seller_hub_*` ids.

**Soft caveats (not blockers):** a new listing enters **"In Review"** (Facebook's
normal automated review for new listings/accounts — still counts as active), and
long-term account survival is unknown and only time answers. Neither changes the
build.

Every technical risk is resolved. The emulator approach is validated.

## Architecture — reuse the OfferUp stack

Everything under `src/automation/android/` is channel-agnostic and is reused as-is:
the `Adb` driver + `parseUiTree` (`adb.ts`), device lifecycle / boot / ADBKeyBoard
/ single Android lock (`device.ts`), and the `waitForNode` / `step` / tracker
helpers.

New for Facebook:

- `src/automation/android/flows/facebook/{post,reprice,relist}.ts` — the three
  flows (mirrors `flows/post.ts` etc. for OfferUp).
- `src/config/facebook.ts` — selector map (the `composer_v3_*` ids, Seller-Hub
  ids, condition-chip labels, `com.facebook.katana`), plus feature flags
  `FACEBOOK_AUTOMATION_ENABLED` and `FACEBOOK_POST_ENABLED` (gate the Post button
  until the flow is hardened, exactly like `OFFERUP_POST_ENABLED`).
- `src/config/facebook-categories.ts` + `docs/facebook-categories.md` — the
  captured taxonomy (see below).
- Server actions in `src/lib/automation-actions.ts`: `postToFacebook(itemId)`,
  `dropFacebookPrice(itemId, listingId)`, `relistOnFacebook(itemId, listingId)` —
  siblings of the OfferUp actions, guarded by the same Android lock.
- Buttons on the Facebook channel card in `ChannelCard.tsx`.

Already in place, unchanged: the Facebook **publisher** (`src/publishers/facebook.ts`,
generates title/body/fields) and the **drop-to-floor task engine**
(`src/lib/tasks.ts`) — the engine already carries Facebook's `renew-then-repost`
relist policy and already emits a `"renew"` vs `"relist"` action for it.

## Category + condition taxonomies (capture first)

Same approach that worked for OfferUp: capture the real data from the app, don't
guess.

- **Categories:** enumerate Facebook Marketplace's full category tree from the
  composer's category picker and commit it to `docs/facebook-categories.md`
  (mirroring `offerup-categories.md`). Category then becomes a **per-item field**
  Wil sets in Dispatch's item form, and the flow selects that exact path — no
  fuzzy mapping, no misc bucket (same decision as OfferUp).
  - Schema: add `facebook_category` (nullable text) to `items` via an additive
    drizzle-kit migration. Capture determines whether Facebook's picker is flat
    or hierarchical; if hierarchical, add `facebook_subcategory` in the same
    migration. (OfferUp is two-level; Facebook's depth is confirmed at capture.)
  - Item form gets a Facebook category picker populated from the taxonomy, beside
    the existing OfferUp one.
- **Conditions:** capture the complete condition set. Observed in the probe:
  **New / Used – Like New / Used – Good / Used – Fair**. A `CONDITION_MAP` maps
  Dispatch's finer scale onto these (e.g. `new→New`, `like_new→Used – Like New`,
  `excellent/good→Used – Good`, `fair→Used – Fair`, `for_parts→Used – Fair`),
  confirmed against the captured list.

## Flows

### Post (hybrid gate — fills, then stops for Wil's tap)

Navigation: Marketplace → **Create listing** → **One item** → composer. Fill
`composer_v3_title`, `composer_v3_price`, `composer_v3_description` (via
ADBKeyBoard for Unicode); attach photos by pushing them to the gallery and
selecting via **Add photos** (same mechanism as OfferUp); select the per-item
**category**; map and select **condition**.

**The flow never taps Publish on a brand-new listing.** It fills everything and
stops at Facebook's review screen for Wil's tap — matching OfferUp, and doubling
as account-safety insurance (a human submitting is the lowest-signal way to post
on a fragile new account). Returns a `posted_review` status.

### Reprice (hands-off)

Your listings → open the listing → **Edit** → replace price → save. Then reconcile
the ledger's recorded price (`syncListingPriceCore`), same as OfferUp. Identify
the listing by title with the ambiguity guard; anchor on the ledger URL where
available.

### Relist (hands-off for renew; review-gated for the 42-day repost)

Facebook's model is friendlier than OfferUp's (which had no native repost). The
engine's Facebook policy is `renew-then-repost` (renew every 7 days, fresh
repost after 42), and it already tags each due task as `"renew"` or `"relist"`:

- **`"renew"` (the common 7-day case):** tap the listing's **Renew** control
  (Seller Hub "Listings to renew" / the per-listing "…" menu). Tiny, low-signal,
  fully hands-off. Stamps `renewedAt`.
- **`"relist"` (the 42-day fresh cycle):** delete the old listing, then post a
  fresh copy, **auto-submitted — fully hands-off** (same as OfferUp's relist).
  The content is identical to a listing that was already reviewed and approved
  when first posted, so there's nothing new to gate on. Rare (every 42 days).

Post-first / delete-last ordering on the 42-day repost, same safety property as
OfferUp: never leave the item with zero live listings.

So the **only** review-gated action in the whole system is a genuinely new
listing (the first time an item is posted). Reprice, the 7-day renew, and the
42-day repost are all hands-off.

## Account-safety guardrails

- `FACEBOOK_POST_ENABLED` gates the Post button until the flow is live-verified.
- **A genuinely new Facebook listing is never auto-submitted** — always the
  review gate (the first post of an item). The 42-day repost auto-submits because
  its content was already approved when first posted.
- **Kill switch:** if Facebook starts checkpointing, flip `FACEBOOK_AUTOMATION_ENABLED`
  off; Dispatch falls back to the existing v1.2 staged-handoff (Firefox + Finder
  + clipboard), which stays intact.
- Human-like pacing between actions (the same deliberate sleeps the OfferUp flows
  use).
- A logged-out / checkpoint state is detected and returned as `login_required`
  rather than blundering forward.
- The Facebook app lives **only on the emulator**, never on Wil's personal
  devices — deliberate, for privacy.

## UI

The Facebook channel card gains the OfferUp-style buttons, gated the same way:
"Post to Facebook" (when `FACEBOOK_POST_ENABLED` and no active listing),
"Sync price to Facebook" and "Relist on Facebook" (when there's an active
listing). Status line reuses the `AndroidResult` rendering.

## Error handling / testing

- Every flow step runs under the existing `step()` wrapper: failure stops the run
  (unknown on-device state) and captures a debug screenshot.
- **Wil-driven live acceptance**, same as OfferUp: Wil drives the first real post,
  reprice, renew, and 42-day repost; each is verified before anything is flipped
  hands-off. Flags stay off until then.

## Out of scope (fast-follows)

- **Buyer-activity reader.** Dispatch reads the Seller Hub's "Chats to answer",
  "Offers need response", and "Listings to renew" counts from the emulator and
  surfaces them on the dashboard (next to OfferUp), so Wil is alerted to respond
  without installing Facebook on his phone. Deferred so v1.4 stays focused on
  post/reprice/relist. (Interim alerting: Facebook email notifications + respond
  via browser.)
- Responding to buyers (stays manual, via browser).
- Multi-item / vehicle / other Marketplace listing types (only "One item").

## Risks

| Risk | Mitigation |
| --- | --- |
| New account checkpointed/banned | Accepted by Wil; review-gated posting minimizes signal; kill switch → assisted flow; app only on emulator. |
| "In Review" listings don't clear to live | Low — the probe listing cleared review and went live within ~1h. Still watch the first posts during acceptance. |
| Category picker deeper/different than OfferUp | Capture-first; schema migration adapts (flat vs. hierarchical) before any flow code. |
| Facebook app update moves selectors | Selectors centralized in `src/config/facebook.ts` — a one-line edit, same as OfferUp. |
