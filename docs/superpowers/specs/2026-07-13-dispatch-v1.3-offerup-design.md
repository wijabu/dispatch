# Dispatch v1.3 ‚Äî OfferUp Automation via Android Emulator

**Date:** 2026-07-13
**Status:** Approved (brainstorming session with Wil, all sections reviewed)
**Builds on:** v1.2 (human-assisted publishing) on main. Publishers and the v1.1 task engine are reused; publishers stay pure.

## Why v1.3 exists

Wil's success bar for Dispatch: remove the friction of (a) drafting listings across apps and (b) the recurring loop of updating each app to drop price or relist. v1.2 solved drafting but left posting/updating manual for the two channels that matter most ‚Äî OfferUp and Facebook. OfferUp has no web posting (app-only) and no personal API, so the only automatable surface is the Android app.

v1.3 automates **OfferUp** through its Android app running in a standard emulator, driven over ADB. Facebook is a deliberate follow-on experiment (see "Not in v1.3").

## Feasibility findings (probe, 2026-07-13)

- **BlueStacks is a dead end.** It gates the ADB control surface (`input`, `push`, `cmd`, `uiautomator`) even with ADB enabled ‚Äî the same hardening that lets it pass OfferUp's integrity checks. Reads work, control does not. See [[bluestacks-adb-deadend]].
- **A standard Android Studio AVD works.** Play-enabled arm64 system image (Android 14) on Apple Silicon. OfferUp installs, logs in (Play Integrity did **not** block it), and functions normally.
- **The AVD control surface is fully open:** `input tap`/`text`, `adb push`, `uiautomator dump`, `pm`, `settings` all work.
- **OfferUp is a React Native app** (`com.horcrux.svg` views), so it exposes clean, semantic `testID`s as resource-ids ‚Äî e.g. `tab-bar-widget.tab.account`, `ProfileListingItem.btn.<title>`, `item-dashboard-screen.edit-post-link`, `item-dashboard-screen.mark-sold-button`. Selector-based automation is viable and robust; no fragile coordinate-tapping needed.
- **OfferUp owner actions** (Manage this item ‚Üí Item Dashboard): **Edit post** (change price/details), **Mark sold**, paid **Promote**. There is **no free one-tap repost/renew** ‚Äî "relisting" on OfferUp means delete + fresh post.

## Design decisions (from brainstorm)

1. **Mechanism: raw ADB + uiautomator, not Appium.** OfferUp's clean testIDs make a heavyweight automation framework unnecessary; a thin driver fits Dispatch's local-first, minimal-dependency ethos (no extra server/daemon).
2. **Submit gate is hybrid.** Price drops and relists run to completion, hands-off. A brand-new listing fills every field and stops at OfferUp's review screen for a one-tap Post by Wil. Dispatch never taps Post on a *brand-new* listing (a relist's repost is the one exception ‚Äî see decision 4 ‚Äî because its content was already reviewed when first posted).
3. **Interaction is hybrid.** The emulator boots windowed but backgrounded. Hands-off flows (reprice/relist) run behind other windows; Dispatch reports "done." A new-post flow foregrounds the emulator window at the review step.
4. **Relist auto-submits its fresh post.** Relisting reuses the post-composition flow, but because the content is identical to the listing that was already live, there is nothing new to review, so the fresh post auto-submits. Only a genuinely *new* listing hits the review gate.
5. **Lifecycle rule (task engine, ALL channels): drop to floor, then relist.** Each item runs one maintenance cadence. While `asking > minimum`, each due cycle is a price drop. Once `asking <= minimum`, drops stop and each due cycle becomes a relist, repeating until sold. This changes v1.1's engine: **relist tasks are suppressed until the item reaches its floor.** Applies to every relist-capable channel (OfferUp, Facebook, Craigslist, Watchuseek, Reddit), not just OfferUp.

## Architecture

A new Android automation surface, sibling to the v1.2 Playwright layer. Publishers stay pure and unchanged; flows consume the same `generate()` output.

```
src/automation/
  browser.ts              ‚Üê existing Playwright singleton (v1.2)
  android/
    adb.ts                thin driver: exec, push, screencap, uiautomator dump+parse (XML
                          -> node tree), find(testId) -> tap / type / read / bounds-center
    device.ts             emulator lifecycle: locate SDK, boot AVD (windowed+backgrounded),
                          wait-for-boot, keep-warm (globalThis singleton), foreground the window
    flows/
      post.ts             new listing -> fill, stop at review (foreground); relist reuses this
      reprice.ts          Item Dashboard -> Edit post -> set price -> save (hands-off)
      relist.ts           post fresh (auto-submit) -> confirm live -> delete old (hands-off)
    types.ts              AndroidFlow contract (same spirit as v1.2 FillScript / FillResult)
    index.ts              registry
```

- The driver keeps flows declarative: `await ui.find("item-dashboard-screen.edit-post-link").tap()`, `await ui.find("Posting.PriceField").type(price)`.
- Each step degrades independently (records which testID failed, screenshots, stops) ‚Äî same philosophy as the v1.2 fill ladder.
- **Server actions** (in existing `src/lib/automation-actions.ts`): `postToOfferup(itemId)`, `dropOfferupPrice(itemId, newPrice)`, `relistOfferup(itemId)`. One flow at a time, guarded by a lock like the Playwright fill lock.

## The three flows

**`post.ts` (new listing ‚Äî review gate):**
1. Acquire lock; ensure emulator booted.
2. `adb push` the item's photos (sort order, primary first) into the gallery + media-scan.
3. Open OfferUp's Sell flow.
4. Select the pushed photos in the picker.
5. Fill title, description, price, category, condition by testID (pickers: select matching option).
6. Advance to the review screen.
7. Foreground the emulator window; return `review` ‚Üí card shows "Review in the window and tap Post, then Mark listed." Dispatch does not tap Post.

**`reprice.ts` (hands-off):**
1. Lock; booted (background).
2. Open your listings (public profile); find `ProfileListingItem.btn.<item title>`; tap; Manage ‚Üí Item Dashboard.
3. `edit-post-link` ‚Üí set new price ‚Üí save.
4. Report done; update ledger.

**`relist.ts` (hands-off, auto-submit):**
1. Lock; booted.
2. Compose and submit a fresh post (reuse `post.ts` composition, auto-submit ‚Äî content unchanged).
3. **Confirm the new listing is live.**
4. **Only then** delete the old listing.
5. Report done; update ledger with the new listing's identity.

## Listing identity

`ProfileListingItem.btn.<title>` testIDs embed the listing title, so reprice/relist match the OfferUp listing to the Dispatch item **by title**. For precision, when an item is posted (or Mark listed), Dispatch captures the OfferUp listing URL into the ledger's existing `url` column as an exact anchor. If a title match is ambiguous, the flow stops and reports rather than acting on the wrong listing.

## Task-engine change (v1.1 `computeTasks`)

- **Price drop due** ‚Äî unchanged trigger, but it is the *only* maintenance action while `asking > minimum`.
- **Relist due** ‚Äî new gate: fires only when `asking <= minimum` (item can no longer drop) AND the channel's relist cadence is due (still respecting `minIntervalDays` / `freshRelistAfterDays`). While the item can still drop, relist is suppressed.
- **Edge case:** an item with no `minimum` set never reaches a floor, so it keeps dropping and never transitions to relist. Documented, not a bug; setting a minimum is what enables the relist phase.
- This is shared logic; all relist-capable channels inherit the drop-then-relist lifecycle.

## Safety / degradation ladder

- Every step wrapped; a failure records the step, screenshots to `data/automation-debug/`, and stops ‚Äî no half-finished action.
- **Relist is post-first, delete-last** ‚Äî the old listing is deleted only after the fresh post is confirmed live, so a failure never leaves the item with zero listings.
- **Reprice** is a bounded edit (price only), safe to run hands-off.
- **Login wall:** if OfferUp is logged out, stop ‚Üí "log into OfferUp in the emulator."
- **Emulator unavailable** (not installed / won't boot): clear error ‚Üí fall back to the v1 copy/paste floor, which never goes away.
- One flow at a time (in-process lock).

## UI integration

- OfferUp's kill switch flips back on. The Publish-tab ChannelCard replaces the "app-only" note with a real **"Post to OfferUp"** button (runs `post.ts`).
- Today's Tasks: OfferUp price-drop tasks get **"Drop price on OfferUp"** (`dropOfferupPrice`); relist tasks get **"Relist on OfferUp"** (`relistOfferup`). Both hands-off, reporting done and updating the ledger.
- Mark-listed / ledger flow otherwise unchanged; a new post ends with Wil tapping Post, then Mark listed (price prefilled).

## Config & setup

- `src/config/android.ts`: SDK path, adb path, AVD name (`dispatch_offerup`), device serial, photo-push dir, OfferUp channel enable flag.
- **Setup is a real dependency**, heavier than v1.2's Playwright: JDK 17, Android command-line tools, a Play-enabled arm64 system image, and the AVD (created 2026-07-13 with `hw.keyboard=yes` and `PlayStore.enabled=yes`). Gets its own setup doc; the emulator must be running (Dispatch boots it on demand and keeps it warm).

## Testing

- **Driver/parse logic** (uiautomator XML ‚Üí node tree ‚Üí find-by-testID ‚Üí bounds-to-tap-point) is unit-tested against real captured dumps ‚Äî fixtures already gathered on 2026-07-13 (home feed, account, public profile, listing detail, Item Dashboard).
- **Flows** need the live app; validated by Wil's live acceptance, same as v1.2. First real post, one reprice, one relist.
- Publishers and the task engine's existing behavior stay green (101 tests); the relist-gating change gets new engine tests (relist suppressed above floor, fires at floor).

## Explicitly not in v1.3

- **No Facebook automation yet.** FB-app-in-emulator is a deliberate *follow-on experiment* after OfferUp ships: higher account risk (FB aggressively flags new-device/emulator logins on personal accounts ‚Üí checkpoint/lockout) and an obfuscated native UI (no clean testIDs). If tried, it aborts the instant FB throws a login checkpoint, and the browser approach stays the fallback.
- No physical-device path (Wil has no Android device; the AVD works, so not needed now).
- No coordinate/screenshot automation unless a specific testID is ever missing.
- No auto-submit of brand-new listings (review gate stays).
- No credential handling ‚Äî the OfferUp session lives in the AVD, logged in by Wil.

## Risk register

| Risk | Posture |
|---|---|
| OfferUp React Native `testID` changes on an app update | Rare (far more stable than coordinates); a flow fix is small and isolated; ladder degrades to copy/paste. |
| OfferUp tightens integrity to block the AVD later | Would fall back to copy/paste; physical device becomes the path if it ever matters. |
| Emulator not running / boot failure | Detected; clear error; copy/paste floor remains. |
| Relist deletes old listing then new post fails | Prevented by post-first-delete-last ordering. |
| Wrong listing matched for reprice/relist | Title match + optional ledger-URL anchor; stop-and-report on ambiguity. |
| Setup complexity (SDK/AVD) | One-time, documented; heavier than Playwright but bounded. |
